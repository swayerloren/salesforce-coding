# Mobile Camera And File Handoff

Salesforce mobile file workflows need explicit runtime handling. Desktop browser success is not enough evidence.

## Review Surfaces

Check every layer that can hide or alter the mobile experience:

- component `js-meta.xml` targets and supported form factors;
- object quick action metadata;
- page layout action placement;
- Dynamic Actions and component visibility filters;
- FlexiPage activation and app/profile assignment;
- permissions, custom permissions, Apex class access, and file access;
- Experience Cloud or external user context when applicable.

For action visibility, use [Quick Action Visibility Matrix](../metadata/quick-action-visibility-matrix.md).

## Mobile Action Bar Rules

- A quick action can exist in source but still be absent from the phone action bar.
- Verify the action type, object, target component, supported form factor, and placement chain.
- Check whether Dynamic Actions override the layout action list.
- Test compact action labels and icon meaning on small screens.
- Avoid action flows that require hover, wide modal layouts, popup windows, or drag-only interactions.
- Preserve a clear cancel path and loading state inside action modals.

## Camera Rules

- Open a visible, correctly sized shell before requesting camera access.
- Treat permission prompts as lifecycle interruptions.
- Confirm the preview element paints after access is granted.
- Stop tracks and clear media state on close, cancel, error, and component disconnect.
- Avoid assuming `renderedCallback` runs only once.
- Keep capture, preview, retake, annotate, upload, and refresh as visible states.

See [Photo Capture, Annotation, Viewer, And Generated File Patterns](../files/photo-capture-annotation-viewer.md) for file lifecycle guidance.

## File Preview And Download Rules

- Prefer platform navigation and preview APIs when they work for the runtime.
- Validate direct download and preview behavior separately on desktop Lightning, Salesforce mobile, and Experience Cloud if those surfaces are in scope.
- Blob URLs and popup tabs are unreliable handoffs in mobile containers.
- A durable file, distribution artifact, or server-authorized URL is safer for mobile preview and external browser handoff.
- Do not put private data, tokens, or raw IDs into a public URL unless the server-side access model is explicit.

## PDF And Report Handoff

- Generated PDFs should become durable artifacts before the UI claims completion.
- Mobile users need a clear path to preview, open externally, download, or return to the record.
- If the PDF is generated by Visualforce, test from the real Lightning action or mobile action context, not only by opening the page URL.
- If the PDF depends on uploaded photos, verify the latest file versions and processing status before generation.

## Chatter And Action UX

- Text insertion menus must survive mobile focus and blur timing.
- Use early pointer selection where blur can close a suggestion list before click.
- Guard duplicate touch and mouse events.
- Keep mention metadata aligned to the final text sent to Apex or ConnectApi.
- Disable submit while posting or uploading attachments.
- Surface recoverable failures without leaking private body text or raw debug payloads.

## Static Validation Checklist

- [ ] Mobile form factor and target metadata inspected.
- [ ] Action visibility chain inspected.
- [ ] Camera/file lifecycle has explicit cleanup.
- [ ] Preview/download behavior is validated or listed as untested.
- [ ] Generated files refresh file lists and parent state.
- [ ] Chatter or action submit cannot double-post.
- [ ] Runtime limitations are reported when real device testing is unavailable.
